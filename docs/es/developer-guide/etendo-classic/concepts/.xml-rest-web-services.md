# Servicios web REST XML

!!! example  "IMPORTANTE: ESTA ES UNA VERSIÓN BETA"

## Visión general

Etendo REST consta de un framework que ofrece servicios de seguridad y de excepciones, y una implementación de servicio web REST de acceso a datos. Los servicios web REST de acceso a datos (DAL) proporcionan un servicio web de tipo CRUD para que las aplicaciones externas puedan recuperar, actualizar, crear y eliminar objetos de negocio mediante solicitudes HTTP estándar.
## Glosario

Este documento utiliza el término objeto de negocio para denotar una entidad y su información dependiente. Un objeto de negocio puede ser una entidad simple, como una divisa, que solo tiene campos primitivos básicos. Por otro lado, también puede ser una estructura de entidades, por ejemplo, una cabecera de pedido con su línea de pedido. Una estructura de objetos de negocio siempre tiene un objeto de negocio que es el propietario de todos los objetos de negocio de esa estructura; por ejemplo, para un pedido de venta, el objeto de negocio de la cabecera del pedido de venta es el propietario de la estructura completa. Otra forma de describir esto es que las líneas de pedido dependen de la cabecera del pedido; es decir, una línea de pedido no puede existir sin su cabecera de pedido y, cuando se elimina una cabecera de pedido, también se debería eliminar la línea de pedido.

## ¿Por qué REST?

Hay una serie de razones por las que un enfoque REST tiene sentido para un servicio centrado en datos: un enfoque REST

- favorece la identificación y el direccionamiento de recursos, lo cual encaja con la naturaleza centrada en datos de las API proporcionadas (un recurso corresponde a un objeto de negocio).
- dispone de acciones (`POST`, `PUT`, `DELETE`, ``GET``) que se corresponden con las acciones CRUD estándar.
- permite enlazar a objetos de negocio específicos o a conjuntos de objetos de negocio. Esta es una funcionalidad muy potente de un enfoque REST y permite una navegación sencilla entre objetos de negocio.
- es sencillo de desarrollar y utilizar, y muy ligero desde un punto de vista arquitectónico.
## Servicio web REST

Es importante tener en cuenta que REST no es un estándar, sino más bien un enfoque o estilo arquitectónico. El REST de Etendo se adhiere al enfoque REST:

- los recursos (objetos de negocio) se identifican mediante enlaces específicos que son estables a lo largo del tiempo
- la acción `GET` no tiene efectos secundarios
- se admiten todas las acciones HTTP: POST, `PUT`, `DELETE`, `GET`
- se utilizan códigos de respuesta HTTP para especificar los resultados de las acciones
## Servicio web REST principal

Una solicitud es procesada por el servlet del servicio web REST (`org.openbravo.service.web.BaseWebServiceServlet). Este servlet utiliza la URL de la solicitud para determinar el tipo de solicitud y cómo gestionarla.

El servlet del servicio web REST también gestiona el inicio de sesión y la autenticación (véase más abajo).

Una solicitud (`GET`, `POST`, `PUT`) puede constar de tres partes:

- la ruta de la URL 
- los parámetros de la URL 
- el contenido de la solicitud 

Una solicitud `GET` no tendrá contenido y está completamente controlada por la ruta de la URL y sus parámetros.

Una solicitud `POST` y `PUT` tendrá contenido, que se asume que es un documento XML válido.

La respuesta de una llamada REST será un documento XML, que contiene los datos solicitados, un mensaje XML de resultado o un mensaje XML de error.
## Servicio web DAL REST

El servicio web DAL REST es una implementación de un servicio web REST en Etendo. Proporciona un servicio web de tipo CRUD para que las aplicaciones externas puedan recuperar, actualizar, crear y eliminar objetos de negocio mediante solicitudes HTTP estándar.

Proporciona la siguiente funcionalidad:

- recuperar un único objeto de negocio usando una URL que contiene tanto el tipo como el ID del objeto de negocio solicitado
- recuperar una lista de objetos de negocio usando solo el tipo, por ejemplo [https://demo.etendo.cloud/etendo/ws/dal/Country](https://demo.etendo.cloud/etendo/ws/dal/Country) (Nombre de usuario: admin / Contraseña: admin)
- consulta, filtrado, paginación y ordenación de listas de objetos de negocio
- actualización de un objeto de negocio existente o de varios objetos de negocio mediante XML y una operación HTTP `POST` o `PUT`
- creación de nuevos objetos de negocio mediante una operación `POST`/`PUT` que envía una estructura XML
- exportación e importación de datos: archivos XML que contienen una mezcla de distintos tipos de objetos de negocio y una mezcla de objetos de negocio nuevos y existentes
- recuperación de una lista de valores de ID e identificador, por ejemplo para rellenar un cuadro de lista en una interfaz de usuario externa
- operación de eliminación usando una URL que apunte a un objeto de negocio específico que deba eliminarse o un documento XML (en el contenido) que contenga objetos de negocio (como XML completo o como XML parcial) que deban eliminarse.

El resto de este documento se centrará principalmente en la funcionalidad de los servicios web DAL REST.
## Solicitud y respuesta única

Una solicitud única es una solicitud que apunta a un único objeto de negocio. Por ejemplo, la URL:

[https://demo.etendo.cloud/etendo/ws/com.smf.securewebservices.obrest/Country/106](https://demo.etendo.cloud/etendo/ws/com.smf.securewebservices.obrest/Country/106) (Nombre de usuario: admin / Contraseña: admin)

devolverá el XML de una instancia del objeto de negocio País con ID = 106 (España). El servicio web REST recuperará este objeto de negocio de la base de datos, creará el XML correspondiente y devolverá el XML resultante. Consulte a continuación una descripción del XML.
## Solicitud y respuesta de consulta

Se utiliza una URL de consulta para recuperar una lista de objetos de negocio. Puede gestionar parámetros de filtro, ordenación y paginación. Un ejemplo de URL de consulta: por ejemplo:

[https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.id='100'&orderBy=name&firstResult=5&maxResult=10](https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.id='100'&orderBy=name&firstResult=5&maxResult=10)
    
(Nombre de usuario: admin / Contraseña: admin)

Esta URL devolverá todos los objetos Country que tengan como moneda el ID = 100 (USD), ordenados por el nombre del país, con la página comenzando en el objeto 5 y devolviendo 10 objetos.

Para consultar campos de fecha para una fecha específica, lo más sencillo es usar un mayor/menor que, ya que los campos de fecha se almacenan en la base de datos con un valor de hora (que puede incluir milisegundos):

[https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.updated%3E'2012-11-20'%20and%20currency.updated%3C'2012-11-21'&orderBy=name&firstResult=5&maxResult=10](https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.updated%3E'2012-11-20'%20and%20currency.updated%3C'2012-11-21'&orderBy=name&firstResult=5&maxResult=10)

(obsérvese que %3E es un > codificado y %3C es un > codificado)

La funcionalidad de consulta REST admite los siguientes parámetros en la URL:

- where: una cláusula where de HQL 
- orderBy: una definición de orderBy 
- firstResult y maxResult: parámetros para soporte de paginación 

El usuario tiene dos tipos de operaciones posibles que pueden realizarse mediante la consulta:

- por defecto, cuando después del nombre de la entidad no hay ningún segmento adicional en la URL: esta operación consultará los objetos de negocio y devolverá (en XML) las propiedades de cada uno de los objetos de negocio (esto también depende de la configuración de seguridad). 
- count: devolverá el número de objetos de negocio que son seleccionables según el filtro. Esta acción se utiliza para dar soporte a vistas paginadas. 

La operación de consulta por defecto devolverá los objetos de negocio (con sus objetos hijo) como XML. Consulte la sección business-object-to-xml. Si solo desea que se devuelvan los objetos sin hijos, entonces puede pasar el parámetro includeChildren (como parámetro de URL) con el valor false.

!!!note
    Al realizar consultas con la cláusula where, cualquier carácter especial debe codificarse en la URL; por ejemplo, el carácter `%` al usar like debe codificarse como `%25`, así:
    [https://demo.etendo.cloud/etendo/ws/dal/Country?where=namelike%20'%25Fr%25'](https://demo.etendo.cloud/etendo/ws/dal/Country?where=namelike%20'%25Fr%25) (Nombre de usuario: admin / Contraseña: admin)
    

!!!note
    Tenga en cuenta que en este primer ejemplo:

    [https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.id='100'&orderBy=name&firstResult=5&maxResult=10](https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.id='100'&orderBy=name&firstResult=5&maxResult=10)
        
    también es posible consultar el servidor usando el identificador de moneda devuelto. No obstante, para este caso debemos usar el nombre original del identificador; en la tabla de moneda es ISO_CODE, por lo que la consulta cambiaría a:
        
    [https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.iSOCode='USD'&orderBy=name&firstResult=5&maxResult=10](https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.iSOCode='USD'&orderBy=name&firstResult=5&maxResult=10)
    

!!!note

    Tenga en cuenta que la cláusula where también puede utilizarse con más de una condición usando la cláusula AND, p. ej.
    
    [https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.id='100'%20and%20hasRegions=true](https://demo.etendo.cloud/etendo/ws/dal/Country?where=currency.id='100'%20and%20hasRegions=true)
## Recuperación de un subconjunto de las propiedades de la entidad
  
Mediante el parámetro `_selectedProperties` es posible especificar la lista de propiedades que el servicio web debe devolver. Este parámetro puede utilizarse tanto en solicitudes individuales como en solicitudes de consulta.

Por ejemplo, para recuperar únicamente el id y el nombre de todos los países:
    
[https://demo.etendo.cloud/etendo/ws/dal/Country?_selectedProperties=id,name](https://demo.etendo.cloud/etendo/ws/dal/Country?_selectedProperties=id,name)
    

La lista de propiedades puede incluir no solo propiedades de la entidad de nivel superior, sino también de sus entidades hijas. Por ejemplo, esta URL recupera el id y el nombre de cada país, y el id y el identificador de cada una de sus regiones:

[https://demo.etendo.cloud/etendo/ws/dal/Country?_selectedProperties=id,name,regionList](https://demo.etendo.cloud/etendo/ws/dal/Country?_selectedProperties=id,name,regionList)
    

También pueden especificarse propiedades de las entidades hijas. Por ejemplo, para obtener adicionalmente el nombre y la descripción de cada una de las regiones del país:

[https://demo.etendo.cloud/etendo/ws/dal/Country?_selectedProperties=id,name,regionList,regionList.name,regionList.description](https://demo.etendo.cloud/etendo/ws/dal/Country?_selectedProperties=id,name,regionList,regionList.name,regionList.description)

El uso del parámetro `_selectedProperties` tiene un efecto beneficioso en el rendimiento: cuanto menos entidades hijas se especifiquen, mejor será el rendimiento de la llamada al servicio web.
## Solicitud y respuesta de actualización/creación

Un objeto de negocio puede actualizarse/crearse mediante REST usando el comando http `POST` o `PUT`. El XML enviado debe ajustarse al esquema XML generado por el servicio web REST. Con una excepción, en situaciones de actualización se permite incluir únicamente las propiedades modificadas en el XML.

El sistema comprobará automáticamente (mediante una consulta) si un objeto de negocio es nuevo o no. Si no se proporciona un ID, entonces siempre se considerará como nuevo; si se proporciona un id, entonces se realiza una consulta a la base de datos. Si el id existe (para ese tipo) en la base de datos, entonces el sistema asumirá una actualización. Si el id no existe, se trata de una inserción, en cuyo caso se mantiene el id (en el xml).

Si la actualización se ha realizado correctamente, se devuelve una respuesta de éxito (`<success/>`). Si falla, se devuelve una respuesta XML de error.
## Solicitud y respuesta de eliminación

El servicio web DAL REST también admite una solicitud de borrado o eliminación. Las solicitudes de borrado utilizan el comando http `DELETE`. La acción de borrado puede aceptar dos tipos de URL:

- un único URI que apunte al objeto de negocio que debe eliminarse, por ejemplo: 

    [https://demo.etendo.cloud/etendo/ws/dal/Product/1000004](https://demo.etendo.cloud/etendo/ws/dal/Product/1000004)

- un URI con un parámetro where: 

    [https://demo.etendo.cloud/etendo/ws/dal/Product?where=name='C02']()

En el segundo caso, es posible eliminar varios objetos en una única solicitud http `DELETE`.
## Actualizar varios objetos de negocio: solicitud y respuesta (Importación)

El sistema también admite la actualización de varios objetos de negocio que se envían en un único archivo. Se deben tener en cuenta los siguientes aspectos:

- el archivo XML puede contener una combinación de objetos de negocio nuevos y existentes (a actualizar)
- puede haber referencias a nuevos objetos de negocio en el mismo archivo XML; estas referencias utilizan un ID único que no necesita estar en la base de datos. Esto significa que la lógica de importación resolverá internamente (dentro del archivo XML) las referencias entre objetos antes de persistir el archivo XML completo.
- el archivo XML puede contener una combinación de distintos tipos de objetos de negocio

Se puede enviar un archivo XML que contenga distintos tipos a la URL genérica del servicio web DAL:

[https://demo.etendo.cloud/etendo/ws/dal](https://demo.etendo.cloud/etendo/ws/dal) (Nombre de usuario: admin / Contraseña: admin)
## Objeto de negocio a XML

La respuesta XML a una solicitud de recuperación consiste en los datos del objeto de negocio o de los objetos de negocio. El objeto de negocio se traduce a XML de la siguiente manera:

- El nombre de la entidad se utiliza como nombre de etiqueta del elemento del objeto de negocio. La etiqueta/elemento de la entidad también tiene atributos para el id y el identificador.
- El nombre de la propiedad se utiliza como nombre del elemento de cada propiedad. Depende de la configuración de seguridad del usuario qué propiedades o información se devuelven (consulte la sección de seguridad).

Los valores de las propiedades se convierten a XML de la siguiente manera:

- El valor de fecha se exporta según el formato datetime de XML Schema.
- El valor de cadena se exporta tal cual (se requiere codificación).
- El valor numérico se exporta usando punto decimal y sin agrupación.
- La referencia (a otro objeto de negocio) se convierte en una etiqueta XML que contiene el id, el nombre de la entidad y el valor del identificador. Esto permite a los consumidores del XML crear fácilmente hipervínculos para recuperar los datos del objeto de negocio referenciado. El identificador puede utilizarse para generar una visualización de la referencia.
- Los objetos hijo (la lista/asociaciones uno-a-muchos) también se exportan y se incrustan en el objeto padre (esto se realiza de forma recursiva a través de la estructura padre-hijo). Puede deshabilitar/evitar la devolución de objetos hijo estableciendo el parámetro (URL) `includeChildren` en `false`.

El elemento utilizado para la versión XML de una referencia también se utiliza en el caso en que sea necesario exportar listas de referencias.
## DAL REST: Visión general de los mensajes de retorno en XML

El diseño analiza distintas funcionalidades. Para resumir, esta sección enumera el conjunto de tipos de retorno que pueden esperarse para diferentes tipos de solicitudes REST:

- una solicitud de un objeto de negocio específico devolverá un documento XML con este objeto de negocio
- una solicitud (una consulta) de una lista de objetos de negocio devolverá un documento XML con los datos de esos objetos de negocio
- una solicitud (una consulta) de una lista de identificadores devolverá un documento XML únicamente con el id, el nombre de la entidad y el identificador de cada objeto de negocio
- una consulta con un parámetro de recuento solo devolverá un mensaje XML simple con un número de recuento `<count>5</count>`
- una solicitud de actualización/inserción devolverá un mensaje XML de éxito `<success/>` si se ha realizado correctamente; el mensaje de retorno contendrá el id del registro insertado.
- una consulta con un parámetro de identificador devolverá una lista de valores de identificador (uno por cada objeto de negocio)

Si las solicitudes fallan, se devuelve un mensaje XML de error.
## Códigos de retorno HTTP

El servicio web REST utiliza los siguientes códigos de error:

- **200 (OK)**: para solicitudes correctas.
- **400 (Bad Request)**: en caso de que el URI no se haya podido analizar o en caso de XML no válido.
- **401 (Unauthorized)**: se devuelve cuando se produce una excepción de seguridad.
- **404 (Not found)**: si el nombre de la entidad no existe o, si se direcciona un objeto de negocio individual, si el objeto de negocio no existe. Tenga en cuenta que una respuesta 410 (Gone) también podría ser aplicable aquí, pero es menos conocida; por lo tanto, se utiliza la 404.
- **409 (Conflict)**: se devuelve para una acción `POST` o `PUT`; especifica que los datos que se han enviado están desactualizados o no son válidos.
- **500 (Internal Server Error)**: si se ha producido un error de aplicación irrecuperable.
## Inicio de sesión y seguridad

El `REST Webservice` proporciona dos métodos para iniciar sesión:

- inicio de sesión con login|/password pasados como parámetros de la solicitud (los nombres de los parámetros son respectivamente l y p) 
- autenticación HTTP básica. La autenticación HTTP básica se puede evitar con el parámetro de solicitud auth=false. En este caso, solo se devuelve el código de estado 401 (Unauthorized).

Los siguientes aspectos son importantes para la seguridad:

- El inicio de sesión y el contexto se almacenan en una sesión por motivos de rendimiento 
- Si el llamador/cliente no tiene sesión (no admite cookies), también es posible incluir la información de inicio de sesión en cada solicitud. **En ese caso, también tiene sentido hacer que la solicitud del servicio web sea sin estado.** Consulte la siguiente sección para obtener más información al respecto. 

Además, el rol por defecto del usuario que está iniciando sesión debe estar habilitado para realizar llamadas a servicios web. Busque "Web Services habilitados" dentro del documento [Rol](../../../user-guide/etendo-classic/basic-features/general-setup/security/role.md#role) para obtener más información.

Tras un inicio de sesión correcto, la acción del servicio web REST se ejecuta dentro del contexto del usuario. El `REST Webservice` utiliza el mecanismo de [control de seguridad DAL](../concepts/data-access-layer.md#security-and-validation) para controlar el acceso de lectura y escritura.
## Solicitudes de Webservice sin estado - Sesión HTTP

!!!note
    Esto es importante para llamadas de webservice de alta frecuencia.  
  
Por defecto, las solicitudes de webservice de OB son con estado. Esto significa que cada llamada de webservice creará y utilizará una sesión HTTP en el servidor de OB. En función de cómo configure el cliente para llamar al webservice, esto incluso puede significar que cada solicitud de webservice crea una nueva sesión HTTP. Esto no es recomendable para solicitudes de webservice de alta frecuencia. En caso de llamadas de alto volumen, puede tener sentido pasar a una implementación sin estado.

Hay 2 formas de conseguir un tratamiento sin estado de las solicitudes de webservice:

- pasar el parámetro `stateless=true` en la URL de la solicitud. Por ejemplo: [https://demo.etendo.cloud/etendo//ws/org.openbravo.howtos.doIt?product=1000001&stateless=true](https://demo.etendo.cloud/etendo//ws/org.openbravo.howtos.doIt?product=1000001&stateless=true)

- anotar la clase del webservice con la anotación `AuthenticationManager.Stateless`: 

    ``` java
    @AuthenticationManager.Stateless
    public class POSTestStatelessWebservice implements WebService {
    ...
    }
    ```

    Al realizar solicitudes sin estado, la implementación del webservice no debe esperar que exista una sesión HTTP ni crear una nueva sesión HTTP. Por ejemplo, el objeto VariablesBase no se puede utilizar en solicitudes de webservice sin estado. Aparte de eso, la mayor parte de la funcionalidad base del framework (como la capa de acceso a datos y el `OBContext object`) está disponible para la lógica.
## Transacción de base de datos

Una solicitud de servicio web REST es básicamente lo mismo que una solicitud HTTP normal. Esto significa que la gestión de transacciones es la misma que para las solicitudes HTTP estándar. Si no se produjo ninguna excepción, entonces la transacción se confirma al final de la solicitud del servicio web (en el servidor de Etendo).
## Campos no actualizables: Cliente, Organización, etc.

Hay una serie de propiedades para cada objeto de negocio que no siempre se pueden modificar a través del servicio web REST:

- **Cliente y Organización**: una actualización mediante servicio web de un objeto de negocio no puede cambiar la información de **Cliente** y **Organización**. En modo de creación, no se puede establecer el **Cliente**; la **Organización** se puede especificar y, si no se especifica, se establece a partir de la información del usuario.
- **Campos de auditoría de actualización y creación**: los campos de auditoría de actualización y creación no son actualizables/modificables a través de la API del servicio web.
## Generación de esquema XSD para el enfoque REST

El servicio web DAL genera un esquema XSD sobre la base del modelo en tiempo de ejecución. Este esquema XSD define el XML que genera el servicio web REST. Esta definición también se utiliza para el XML empleado para actualizar Etendo mediante servicios web REST.

Para una instancia de Etendo en ejecución en `https://demo.etendo.cloud/etendo`, el XSD se puede obtener mediante esta solicitud: [https://demo.etendo.cloud/etendo/ws/dal/schema](https://demo.etendo.cloud/etendo/ws/dal/schema) (Nombre de usuario: admin / Contraseña: admin)
## Gestión de errores

Cuando una solicitud no puede procesarse y se produce un error, se devuelve un documento XML de error y el código de respuesta HTTP correspondiente. El documento XML de error contiene el mensaje de la excepción. La traza de pila y otra información más detallada no se envían intencionadamente (por motivos de seguridad).
## Plantillas XSLT sencillas

Etendo dispone de algunas plantillas XSLT sencillas para mostrar cómo se puede procesar el XML y demostrar que REST es un buen enfoque para permitir la navegación a través de conjuntos de datos. La siguiente URL muestra la URL de inicio para una llamada a un servicio web DAL utilizando las plantillas XSLT:

- [https://demo.etendo.cloud/etendo/ws/dal/?template=types.xslt](https://demo.etendo.cloud/etendo/ws/dal/?template=types.xslt) (Nombre de usuario: admin / Contraseña: admin)
- [https://demo.etendo.cloud/etendo/ws/dal/Country/106?template=bo.xslt](https://demo.etendo.cloud/etendo/ws/dal/Country/106?template=bo.xslt) (Nombre de usuario: admin / Contraseña: admin)
- [https://demo.etendo.cloud/etendo/ws/dal/Currency/100?template=bo.xslt](https://demo.etendo.cloud/etendo/ws/dal/Currency/100?template=bo.xslt) (Nombre de usuario: admin / Contraseña: admin)
    
Las plantillas XSLT se pueden encontrar en el paquete `org.openbravo.service.rest`.

Los casos de prueba REST se pueden encontrar en el proyecto de desarrollo de Etendo en la carpeta `src-test` y, a continuación, en el paquete `org.openbravo.test.webservice`. Los casos de prueba REST ofrecen una primera impresión de cómo realizar solicitudes REST directamente desde Java, incluida la autenticación básica.
